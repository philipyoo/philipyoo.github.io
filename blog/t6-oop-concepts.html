Ruby Classes vs Modules


Posted on July 1, 2015

Previous Post | Next Post

Ruby Classes are convenient ways to group together multiple methods and constants. They include many unique objects that aren't required to use, but are there for convenience.

There is the initialize method, which is unique compared to other methods you may run into. You can only have one of this initialize method per class, and its main purpose is to run whatever is within the initialize method before you run anything else in the class. So just like its name, it initializes the instance of the class.

There are attr reader and attr writer methods. These are more just syntactic sugar. Rather than typing out the full method, you are provided a short-cut way of creating these methods. Just like their names imply, reader methods provide the ability to read instance variables and their current values. Writer methods provide the ability to change the value of an instance variable outside of the class. There is also the attr accessor method which is another syntactic sugar, and will combine both reader and writer methods into one.

Why would we want to group methods together?

Well, imagine you had thousands of lines of code. It's going to literally become a nightmare trying to track things down. You need to figure out which message each method is sending, what variables are currently set to, you will probably have hundreds of different variables everywhere, and you will probably have hundreds of different (some repetitive) names. These are just a few of the problems you will likely run into.

Classes are convenient for organization. Although classes are objects themselves, they also create other objects. These objects that they create, we typically call them instances of the class. If we had a class called `Ninja`, we would probably see methods like `throw_ninja_star`, `use_sword`, `disappear`, etc. When we create an instance of this class, we would do something like: `philip = Ninja.new`. In this case, `philip` is now a new instance of class Ninja, and it has the ability to call on many of the methods in that class. Also, instance of `philip` is separate form instances of other `Ninja`s.

What are Ruby Modules?

Ruby Modules may seem similar to classes, except they can't create instances of the module and they can't create new objects. Modules are typically used to store classes, methods, and constants, and are really great for organization purposes as well.

Modules are what they call a mixin in Ruby. Ruby is an object-oriented programming language, meaning most everything in Ruby is considered an object. But, Ruby is a single inheritance language, whereas many other object-oriented languages are multiple inhertiance languages. So in Ruby, objects can inherit from another object (i.e. Ninja class can inherit from Person class) and take in all of the objects and be able to use them. However, objects cannot inherit from more than one object.

This is where Modules come in handy for Ruby. Rather than being restricted from what you inherit from, you have the ability to include different Modules (mixin the Module) so that your objects can not only inherit from another object, but they can also include different types of properties provided by Modules.

Mentioned before, Modules can store classes. This is a convenience you will see when your code gets really long. Since we can store classes within Modules, this brings about a topic called Namespacing.

Basically, you are going to have tons of different method names, variable names, constants, etc. Namespacing means that even though you may use the same names, whether it be a method, variable, or constant, placing these same names within a module allows them to live separately from their shared name in say another module or class or globally.

As an example, say you have a module called `Clan` with a constant inside it called `Stark`. Also, you have a global constant in your program (outside of modules `Clan`) called `Stark` as well. Calling on the global constant, you would just need to type out `Stark`. Calling on the constant inside the module, you would need to type out `Clan::Stark`. Both of these constants may share the same name, but they are different objects.
